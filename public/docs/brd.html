1. Executive Summary

SMSaaS is a Software-as-a-Service bulk SMS platform designed for the Maldivian market. The platform enables local businesses to send mass SMS messages to their customers without requiring any technical knowledge, Twilio account, or third-party setup.

The platform operator builds and maintains a single centralised SMS API server. Business clients are onboarded as Tenants with isolated accounts and dedicated console interfaces. Revenue is generated through a credit-based SMS resale model ‚Äî buying wholesale from Twilio and selling at a markup to clients.

Scope of this document

This BRD defines what the business needs the system to accomplish and why. It does not define how features are implemented ‚Äî that is covered in the Functional Requirements Document (FRD).

2. Problem Statement

Maldivian businesses that want to send bulk SMS to their customers currently face significant barriers:

Current pain points

Pain PointImpact on Business
Foreign SMS providers require business verificationComplex, time-consuming registration process ‚Äî often abandoned
Technical integration requiredMost local businesses have no technical staff ‚Äî APIs are inaccessible
Manual SMS sendingImpossible at scale ‚Äî sending 1,000 messages manually is not viable
No local dedicated SMS platformBusinesses resort to WhatsApp broadcasts or social media ‚Äî inferior reach and professionalism
Lack of delivery trackingNo way to know if messages arrived ‚Äî no campaign intelligence
High per-message cost from resellersInformal SMS services are expensive and unreliable

Opportunity

There is no dedicated, simple, self-service bulk SMS platform built for Maldivian businesses. The market is underserved. A platform that removes all friction ‚Äî no registration with Twilio, no technical knowledge, no manual process ‚Äî addresses a real and immediate need.

3. Business Objectives

Primary objectives

BO-01
Generate recurring passive revenue

Build a platform that generates ongoing income through SMS credit markup, with minimal operator involvement after initial client setup.

BO-02
Provide a genuinely useful service to Maldivian businesses

Solve the real problem of bulk SMS accessibility in the Maldives. The product must work reliably and be simple enough for non-technical staff to operate.

BO-03
Build a scalable multi-tenant architecture

The platform must support multiple business clients from a single server instance. Adding a new client must not require significant additional infrastructure.

Secondary objectives

BO-04
Establish the operator as a trusted SMS infrastructure provider

Position the platform as the go-to SMS solution for Maldivian businesses, building a defensible client base over time.

BO-05
Maintain positive cash flow from day one

The credit pre-purchase model means clients pay before they send. The platform should never be in a position where it has delivered SMS without payment.

5. Scope

In scope ‚Äî Version 1.0

  • Multi-tenant SMS API server with full tenant isolation
  • Contact management ‚Äî create, import (CSV), update, delete
  • Group management ‚Äî create groups, add/remove contacts
  • Bulk SMS campaigns ‚Äî send to a group, track progress in real time
  • Credit system ‚Äî pre-purchase model, automatic deduction, refunds on failure
  • Admin panel ‚Äî create tenants, top up credits, rotate API keys, disable accounts
  • Twilio webhook integration ‚Äî real-time delivery status updates
  • Client console ‚Äî Next.js frontend per client, connects via API key
  • Background worker ‚Äî BullMQ + Redis, 3 retries, concurrency controls
  • API documentation ‚Äî hosted at /docs

Out of scope ‚Äî Version 1.0

  • Scheduled / future-dated campaigns
  • Two-way SMS (inbound message handling)
  • Message personalisation / merge tags
  • Self-serve credit purchase (payment gateway integration)
  • International phone number support
  • Email or WhatsApp channels
  • Multi-user access per tenant with roles
  • Analytics dashboards beyond campaign progress

8. Risks & Assumptions

Risks

RiskLikelihoodImpactMitigation
Twilio rate increase to MV numbersMediumMediumBuild adequate margin buffer. Review rates quarterly. Pass increases to clients with notice.
Twilio account suspensionLowHighFollow Twilio's usage policies strictly. Implement opt-out handling. Maintain account in good standing.
Maldivian carrier delivery issuesMediumMediumTest both Dhiraagu and Ooredoo before go-live. Monitor delivery rates per carrier.
Client uses platform for spamLowHighRequire clients to confirm they have permission to contact their list. Enforce opt-out compliance.
Redis or Postgres outage on RailwayLowHighRailway managed databases have built-in redundancy. Worker retries prevent message loss.
Slow client adoptionMediumMediumDevelopment fee provides immediate revenue. Credit model means no ongoing cost if clients are inactive.

Assumptions

  • Twilio can deliver SMS to both Dhiraagu and Ooredoo MV numbers reliably
  • Target clients have a contact list ready to import (or can compile one)
  • Clients are comfortable with a web-based tool ‚Äî no mobile app required for v1
  • The platform operator has or can obtain a Twilio account with MV routing enabled
  • Railway provides sufficient uptime and performance for the expected load
